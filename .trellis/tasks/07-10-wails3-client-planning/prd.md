# 规划 Wails 3 客户端兼容架构

## Goal

在完整理解 Gist 现有产品能力与代码边界的基础上，规划一个基于 Wails 3 的桌面客户端演进方案，并确保现有功能及既有使用方式不发生未经确认的回归。

## Requirements

- 梳理当前项目背景、用户可见功能、前后端架构、运行方式以及关键依赖。
- 基于 Wails 官方资料研究 Wails 3 的能力、成熟度、平台约束与集成方式。
- 明确“现有功能不受影响”的可验证范围，包括功能、接口、数据、部署和使用方式等兼容维度。
- 在现有代码基础上设计 Wails 3 客户端，优先复用已经验证的业务能力，避免无依据地重写。
- 桌面客户端必须同时支持本地模式与远程模式；二者都是正式能力，不把其中一种降级为开发专用路径。
- 远程模式允许连接用户已经部署的 Gist 服务端并使用该服务端的数据。
- 本地模式无需预先部署 Gist 服务端，应用在本机提供完整 Gist 能力并使用本机数据。
- 本地模式与远程模式的内容资料库相互独立；切换模式只切换当前资料库，不复制或同步数据。
- 订阅、文章、已读状态、收藏、设置和 AI 缓存均归属于各自内容资料库；一方的修改不得隐式影响另一方。
- 网络代理设置归属于对应内容资料库，并且只控制承载该资料库的 Gist 后端访问外部资源。
- 桌面客户端只提供一套网络代理设置界面，并按当前模式把读取、测试和保存操作直接路由到对应服务。
- 本地模式直接读写桌面进程内本地服务的网络代理设置；远程模式在线直接读写远程服务的网络代理设置，不在本地资料库中保存远程设置副本。
- 切换模式时必须重新读取目标服务的网络代理设置，不得在本地与远程之间复制、合并或自动同步配置。
- 远程服务不可用或保存失败时，远程网络代理设置必须保持未修改状态；客户端不得提供离线修改队列，也不得把未保存值误显示为远程已生效配置。
- Wails 桌面外壳、桌面客户端到远程 API 的连接以及 GitHub 更新检查不得读取、继承或修改任一内容资料库的网络代理设置；首版不提供单独的桌面连接代理配置。
- 桌面客户端只维护一个固定的本地资料库，并且最多保存一个远程服务端连接。
- 首版不支持多个本地资料库、多个远程服务端配置或跨资料库聚合视图。
- 桌面客户端必须先确定进入本地模式还是远程模式，再启动对应模式的认证流程。
- 首次进入本地模式时沿用现有单用户注册流程，之后使用该本地用户的凭据登录。
- 进入远程模式时只向已配置的远程服务端认证，并使用该远程服务端的用户凭据。
- 本地与远程的凭据、认证令牌和登录状态必须分别保存，不得跨模式复用。
- Windows 桌面版必须把本地认证令牌和远程认证令牌保存为当前用户 Windows 凭据管理器中的两个独立凭据；远程凭据必须绑定规范化后的远程服务地址。
- Windows 桌面版不得把认证令牌持久化到 WebView `localStorage`、SQLite 设置或普通文件；WebView 只持有当前运行期间所需的会话状态，由桌面传输层为请求提供对应模式的令牌。
- 本地或远程用户密码不得由桌面客户端持久化；Windows 凭据管理器不可用或写入失败时，不得回退到明文存储，最多维持当前进程的内存会话并明确提示下次需要重新登录。
- 本地登出只删除本地令牌，远程登出只删除当前远程令牌；任一操作不得改变另一模式的认证状态。
- 现有 Web/PWA 继续使用当前浏览器认证存储和 API 行为，不得因桌面凭据适配而改变。
- 启动或切换到某个模式时，如果该模式存在已保存令牌，客户端必须先验证令牌并在有效时自动进入，不得再次索取密码。
- 远程令牌验证前必须先完成远程服务地址安全校验和 API 兼容性握手，验证流程不得向未经确认或不兼容的目标发送令牌。
- 只有对应后端明确返回认证令牌无效、过期或被撤销时，客户端才清除该模式令牌并显示该模式的登录页。
- 网络不可达、超时、证书错误、API 不兼容或服务端临时故障不得清除已保存令牌，也不得被误报为凭据错误。
- 桌面客户端首次启动时必须先显示模式选择。
- 首次选择后，桌面客户端记住上次使用的模式，后续启动默认尝试进入该模式。
- 应用内必须提供显式模式切换入口；切换成功后更新下次启动使用的模式。
- 主界面必须持续显示当前内容资料库来源，并且该标识不得由用户隐藏；本地模式显示“本地”，远程模式显示规范化后的远程主机、显式端口和路径前缀。
- 内容资料库来源标识同时作为模式切换入口，但在候选模式完成验证并切换成功前必须继续显示当前来源，禁止提前显示未生效的目标。
- 远程服务不可用、认证失效或 API 不兼容时仍必须显示对应远程来源；HTTP 远程连接还必须同时保留既定的不安全标识。
- 来源标识必须使用可读文本表达模式，不能只依赖颜色、图标或悬停提示区分本地与远程。
- 当启动模式为远程模式但远程服务不可达、地址失效或网络不可用时，客户端必须停留在远程模式错误状态。
- 远程模式错误状态必须提供重试、修改远程连接和显式切换到本地模式；禁止自动回退到本地资料库。
- 第一个可交付桌面版本只支持 Windows 10/11 x64。
- 首版不承诺 Windows ARM64、macOS 或 Linux 桌面包，但代码边界不得无理由阻碍未来扩展；现有 Linux Docker/Web 交付继续独立维护。
- Windows 首版的唯一正式桌面交付物是 NSIS 安装包；免安装便携包不进入正式支持矩阵。
- NSIS 只为当前 Windows 用户安装，默认程序目录为 `%LOCALAPPDATA%\Programs\Gist`；首版不提供所有用户或计算机级安装。
- 首版固定使用 `%LOCALAPPDATA%\Programs\Gist` 作为程序目录，不提供安装目录选择、命令行改写或安装后的程序目录迁移。
- 覆盖升级、修复、自动升级回滚和卸载必须定位同一固定程序目录，且不得把该目录策略扩展到用户数据目录。
- 正常安装、覆盖升级和卸载不得要求管理员权限，也不得创建计算机级自启动项、注册表项或快捷方式；受 Windows 策略限制而无法写入时必须明确失败，不能静默改为计算机级安装。
- 程序目录与 `%LOCALAPPDATA%\Gist\data` 用户数据目录必须保持独立的生命周期和清理边界。
- 桌面客户端在后台定期检查新版本；检查失败不得阻断启动或本地/远程功能。
- 发现新版本时必须先提示用户，只有用户明确确认后才下载并启动新的 NSIS 安装包。
- 首版禁止静默下载后安装，也不使用 Wails updater 直接替换当前程序文件。
- 用户首次确认更新后可以下载并验证 NSIS 安装包，但该确认不授权客户端取消前台任务或立即退出。
- 只要仍有前台任务，客户端就必须阻止启动已下载的安装器；任务全部结束后必须再次询问用户是否立即安装。
- 只有用户第二次明确确认后才能执行优雅退出并启动安装器；暂缓或拒绝安装不得取消任务、退出客户端或损坏已验证的待安装状态。
- 已保存远程连接的 API 主版本即使不在目标桌面版本的支持范围内，也不得阻止用户下载或安装该桌面更新。
- 桌面更新完成后，仅在用户进入远程模式或重试远程连接时执行兼容性握手并提示不适配；本地模式不得因此受限。
- 升级后的远程 API 不适配不得自动删除远程连接、远程令牌或 HTTP 风险确认；远程服务升级到兼容版本后，用户可以重试原连接。
- 启动升级安装前必须完成当前应用的优雅退出；覆盖升级和卸载不得删除本地资料库或远程连接配置。
- 启动覆盖升级前必须创建受保护的升级恢复点，并保留当前正式版本以及升级前的一致性本地资料库快照。
- 新版本首次启动必须先完成健康检查；只有应用能够正常启动、资料库迁移成功且迁移后完整性检查通过，升级才视为成功。
- 如果新版本首次启动失败、资料库迁移失败或迁移后完整性检查失败，必须自动恢复上一正式版本和升级前的本地资料库，不得让用户进入部分升级状态。
- 自动升级回滚只处理紧接升级发生的首次启动或迁移失败；升级验证成功并进入正常使用后，不提供任意版本降级或数据倒退能力。
- 自动回滚失败时必须保留升级恢复点，停止继续启动或迁移，并向用户提供明确的人工恢复信息，禁止删除仍可用于恢复的数据。
- 正式 NSIS 安装包必须具有有效的 Windows Authenticode 签名。
- 更新清单和安装包必须通过客户端内置公钥进行密码学签名验证；Authenticode 或应用内签名任一验证失败都必须拒绝启动安装。
- 测试构建与正式更新通道必须隔离，未签名测试产物不得被正式客户端识别为可安装更新。
- 发布设计必须包含私钥保管、CI 签名权限、签名失败关闭和公钥轮换方案。
- 桌面客户端、服务端/Docker 与 Web/PWA 使用统一的 Gist 产品版本号和同一个稳定 Git Tag，不建立彼此独立的组件版本线。
- 每个稳定 Release 必须同步产出该版本的签名 NSIS、签名更新清单、服务端/Docker 与 Web/PWA 交付物；任一必需产物、测试或签名失败时不得发布部分稳定版本。
- 所有正式产物必须能够追溯到同一提交和版本号；桌面客户端展示的产品版本、服务端报告的产品版本及 Release 元数据必须一致。
- 正式客户端只从项目现有 GitHub Releases 检查更新，并且只接受不带预发布标识的稳定版本。
- 签名更新清单与 NSIS 安装包必须作为同一稳定 Release 的一致性资产发布。
- 首版不提供预览/测试通道切换，也不建设独立更新服务。
- 启动后仅在距上次自动检查已超过 24 小时时执行后台更新检查；检查不得阻塞启动。
- 应用持续运行时每 24 小时自动检查一次；设置中的手动检查始终可用且不受自动检查间隔限制。
- 客户端隐藏在托盘或由 Windows 登录自启动进入后台时，发现新版本只能发送 Windows 系统通知并标记托盘更新状态，不得主动显示或抢占主窗口焦点。
- 用户点击更新通知或主动打开主窗口后，客户端才显示更新确认；通知和托盘标记本身不得触发下载或安装。
- Windows 通知被系统禁用或发送失败时，客户端必须保留待处理更新状态，并在用户下次打开主窗口时显示更新提示；同一版本的重复检查不得堆叠重复通知。
- 桌面版的本地资料库固定存放在当前 Windows 用户的 `%LOCALAPPDATA%\\Gist\\data`，首版不提供自定义路径或目录迁移。
- 现有服务端继续使用 `./data`、`GIST_DATA_DIR` 和 `GIST_DB_PATH`，不得因桌面目录策略改变其行为。
- 桌面本地模式不得自动扫描、复用或直接打开现有服务端的 `./data` 或自定义数据目录，同一 SQLite 资料库不得由桌面与服务端共享。
- 用户可通过远程模式原样使用已有服务端资料库；只有用户主动执行完整备份与恢复时，数据才会转入独立的桌面本地资料库。
- 首版不提供直接复制活动数据库文件、自动导入常见目录或服务端到本地的隐式迁移。
- NSIS 卸载默认保留本地资料库、桌面设置、远程连接和保存的认证状态。
- 卸载界面必须提供默认不勾选的“删除所有 Gist 用户数据”复选框，并在执行前列明将删除的内容和数据目录。
- 只有用户明确勾选时，卸载才删除本地资料库、桌面设置、远程连接、保存的令牌及相关桌面数据；清理必须限定在 Gist 自有目录和凭据范围内。
- 本地模式必须提供由用户主动触发的完整备份与恢复；首版不提供自动定期备份。
- 完整备份至少覆盖本地订阅、文章、阅读状态、收藏和本地设置，并与只导出订阅结构的 OPML 明确区分。
- 完整备份不得包含远程资料库；远程数据的备份和恢复始终由远程服务端负责。
- 备份必须基于一致性快照，恢复前必须验证格式、完整性和兼容版本，不能在数据库使用期间直接复制活动文件。
- 本地完整备份必须包含可完整还原的本地账号、AI/代理凭据、AI 缓存、图标及其他本地状态，但不得包含远程连接或远程认证令牌。
- 每次创建完整备份都必须由用户设置独立备份密码；应用不得保存、同步或提供找回该密码的能力。
- 备份必须使用可跨 Windows 重装或设备恢复的密码加密与认证方案，不能只绑定当前 Windows 用户或设备。
- 错误密码、密文篡改、内容损坏或版本不兼容均必须在修改当前本地资料库前失败关闭。
- 恢复流程在完成解密、完整性和版本验证后，必须明确提示当前本地资料库将被整体替换，并再次取得确认。
- 替换前必须创建受保护的临时回滚快照并安全停止本地调度、流式任务和数据库访问；成功后重新打开恢复后的本地资料库。
- 替换或重新打开任一步失败时必须自动恢复原资料库；首版不提供备份与当前资料库的合并恢复。
- 本地完整恢复不得修改远程连接、远程认证状态或远程资料库。
- 用户请求模式切换时，如果当前资料库存在 AI 流、OPML 导入等用户发起且仍在进行的前台任务，客户端必须阻止立即切换。
- 阻止切换时必须提供“取消任务并切换”和“留在当前模式”；禁止无确认地隐藏或取消前台任务。
- 定时刷新等资料库后台维护不阻塞模式切换，并继续遵守各自模式的后台规则。
- 对已经交给远程服务端且无法撤回的任务，客户端必须在允许切换前明确说明任务仍可能在远程继续，不能伪装成已取消。
- 主窗口隐藏到托盘时，用户主动发起的 AI、OPML 等前台任务完成或失败必须发送 Windows 系统通知；通知只显示任务类型和结果，不得包含文章正文、凭据或其他敏感内容。
- 用户点击前台任务通知时必须恢复已有主窗口，并在对应资料库仍可进入时返回相关任务页面；无法恢复原页面时应显示该任务的明确结果，而不是静默丢弃通知。
- 主窗口可见时只更新应用内任务状态，不重复发送系统通知；定时刷新等后台维护无论窗口是否隐藏都不得发送任务完成通知。
- Windows 通知不可用时，任务结果必须保留在应用内，并在用户下次打开主窗口时可见。
- 本地模式与远程模式始终使用桌面客户端安装包内置的同一套前端；远程服务端只提供 API、能力声明和远程资料库。
- 桌面客户端不得在 Wails 窗口中直接加载远程服务端托管的 Web/PWA 页面，也不提供在内置前端与远程页面之间切换的选项。
- 现有服务端继续独立提供其 Web/PWA 前端；桌面内置前端的发布和更新不得替换或禁用该路径。
- Wails 主窗口和任何桌面子窗口只允许导航到安装包内置应用资源与应用内部路由，不得加载外部网页。
- 用户明确点击文章原文、订阅站点或正文中的 HTTP/HTTPS 外链时，桌面客户端必须交给 Windows 默认浏览器打开，不得在 Wails WebView 内导航或创建承载外部网页的新 WebView。
- 非用户触发的外部导航以及 `javascript:`、`data:`、`file:` 等未获支持的外部协议必须被阻止；打开系统浏览器失败时必须保留当前界面并显示错误。
- 现有 Web/PWA 继续使用浏览器新标签行为，不得因桌面外链适配而改变。
- Windows 桌面版的 OPML 导入/导出、本地完整备份/恢复和诊断包导出必须使用系统原生打开或保存对话框，不得依赖 WebView 文件输入、Blob 下载或浏览器下载目录。
- 用户取消原生文件对话框时不得产生错误、创建空文件或修改当前资料库；选择文件后仍必须执行对应格式、完整性和兼容性验证。
- 远程模式导入时只向远程后端上传用户选中文件的内容，不得向远程服务暴露或保存本机文件路径；远程导出则先取得内容，再由桌面客户端写入用户选择的位置。
- WebView 不得获得对用户未选择路径的任意文件系统访问能力，也不得持久化最近文件内容；现有 Web/PWA 继续使用浏览器文件输入和下载流程。
- 远程模式不提供持久化离线资料库、离线文章浏览或离线写入队列。
- 远程服务不可用时不得用缓存内容伪装成可用的远程资料库；用户只能重试、修改远程连接或显式切换到独立的本地模式。
- 本地模式的服务能力只供桌面客户端进程内部使用，不监听回环或局域网端口，也不允许浏览器或其他设备访问本地资料库。
- 浏览器或局域网访问继续由现有 `cmd/server`/Docker 交付负责；桌面本地模式不得成为第二种对外服务端。
- 本地 AI/导入等流式能力必须由桌面内部传输适配层承载，不能通过隐藏的 localhost HTTP 服务规避 Wails AssetServer 限制。
- Wails 3 虽由上游标记为 Alpha，Gist 桌面版仍按正式产品版本开发、签名、发布和支持，不使用 Preview 标签，也不等待 Wails 3 稳定版。
- 接受上游预发布状态不等于跟随无锁定版本；正式构建仍必须精确锁定并验证 Wails CLI、Go module 和前端 runtime 的兼容版本组合。各组件版本号可以使用上游各自的版本线，但任何一项都不得使用 `latest`、通配范围或未锁定解析结果。
- 每次 Wails 升级必须经过完整桌面回归和安装/更新验证后才能进入正式 GitHub Release。
- 只要桌面客户端进程仍在运行，本地资料库就继续执行现有定时刷新，即使当前使用的是远程模式。
- 远程资料库的后台刷新始终由远程服务端负责；桌面客户端不得为远程资料库再启动一套本地调度器。
- 每个 Windows 用户同一时间只能运行一个 Gist 桌面客户端进程；该实例独占本地资料库迁移、访问和后台调度职责。
- 再次启动 Gist 时不得创建第二个应用实例，而应唤醒并显示已有主窗口；已有窗口隐藏在托盘或最小化时也必须恢复到前台。
- 单实例判定必须能够识别并清理进程异常终止留下的陈旧状态，不能永久阻止后续正常启动。
- 用户点击窗口关闭按钮时，桌面客户端必须先让用户选择“完全退出”或“隐藏到系统托盘”，不得直接执行其中一种。
- 选择完全退出时必须优雅停止本地调度和后台任务、关闭本地资源并终止进程。
- 选择隐藏到系统托盘时应用继续运行且本地资料库继续刷新；托盘必须提供重新打开主窗口和完全退出入口。
- 从窗口关闭对话框、系统托盘或其他入口请求完全退出时，如果仍有前台任务，客户端必须再次列明影响并要求确认，不得直接终止任务。
- 只有用户明确选择“取消任务并退出”后，客户端才取消可取消的前台任务并执行完全退出；用户放弃退出时必须保持任务和当前界面不变。
- 隐藏到系统托盘不取消前台任务，任务必须继续运行；对已经交给远程服务且无法撤回的任务，退出确认必须如实说明其仍可能在远程继续。
- 关闭选择对话框必须提供“记住我的选择”；启用后，后续关闭按已记住的行为执行而不再询问。
- 记住的关闭行为是桌面客户端全局偏好，不随本地/远程模式切换；设置中必须能够查看并重置为每次询问。
- 设置中必须提供当前 Windows 用户级别的“登录 Windows 后自动启动”开关，并且新安装时默认关闭；安装器不得替用户开启。
- 用户开启登录自启动后，客户端在 Windows 登录时静默启动到系统托盘，不主动显示主窗口，并立即启动本地资料库的既有后台维护。
- 登录自启动不得自动进入远程认证界面或因远程连接不可用而弹出窗口；用户从托盘打开主窗口后再按启动模式进入对应流程。
- 用户关闭登录自启动后必须移除由 Gist 创建的自启动项；覆盖升级保留用户选择，卸载则清理由 Gist 创建的自启动项。
- 首版不得采集或自动上传使用遥测、诊断事件、日志或崩溃报告，也不得内置默认启用的第三方遥测 SDK。
- 桌面客户端只在 `%LOCALAPPDATA%\Gist\logs` 保留有界轮转的本地诊断日志；日志写入前必须移除密码、令牌、AI/代理密钥、备份密码、文章正文及其他可用于恢复凭据或内容资料库的数据。
- 本地诊断日志最多保留最近 14 天，并且日志目录总量不得超过 50 MiB；任一上限触发后必须从最旧日志开始清理。
- 诊断包只能由用户主动导出到其选择的位置；导出前必须列明包含的应用版本、Windows/WebView2 环境摘要和脱敏日志，不得包含本地资料库、完整备份、远程令牌或文章内容。
- 生成诊断包不得触发任何网络上传；是否以及如何把该文件交给支持人员完全由用户决定。
- 用户主动导出到自选位置的诊断包不属于日志目录轮转范围，客户端不得在之后自动删除该文件。
- 远程连接优先使用 HTTPS，但允许用户连接未加密的 HTTP 服务，以兼容现有局域网和 NAS 部署。
- 首版远程连接只能由用户手动输入完整服务地址；客户端不得扫描局域网，也不提供 mDNS、广播发现、二维码配对或自动候选服务列表。
- 手动输入的地址必须先经过地址规范化、安全校验和兼容性握手，验证成功后才能保存为远程连接；发现其他 Gist 服务不得隐式替换用户指定的目标。
- 远程服务地址只接受 `http` 或 `https`，必须包含主机，可以包含显式端口和反向代理路径前缀，例如 `https://example.com/gist`。
- 远程服务地址不得包含用户名、密码、查询参数或 URL 片段；出现这些内容时必须拒绝保存，而不是静默丢弃后继续连接。
- 客户端必须使用结构化 URL 解析和连接规则规范化尾部斜杠，并在保留路径前缀的基础上定位兼容性握手与 `/api` 端点，禁止使用易产生双斜杠、路径逃逸或主机替换的字符串拼接。
- HTTP 风险确认和已验证兼容性必须绑定到规范化后的完整远程服务地址；协议、主机、端口或路径前缀任一变化都视为新地址并重新验证。
- 远程请求只能在协议、主机、端口均不变且目标仍位于已确认路径前缀内时自动跟随有界重定向；重定向循环或超过上限必须失败并显示明确错误。
- 重定向导致协议、主机、端口或路径前缀任一变化时，客户端必须停止跟随、显示目标，并且只有用户将其确认为新的远程服务地址后，才从安全校验和兼容性握手重新开始。
- 携带远程令牌或其他凭据的请求绝不得自动跨远程服务地址重定向；HTTPS 降级为 HTTP 时还必须重新执行 HTTP 明文风险确认。
- 在保存 HTTP 远程连接或向其发送任何凭据前，客户端必须明确说明明文传输风险并取得用户确认。
- 已确认的 HTTP 连接必须持续显示不安全标识；远程地址发生变化或连接被移除后不得沿用旧确认。
- HTTPS 远程连接必须通过 Windows 系统信任链验证；自签名未受信、过期、主机名不匹配或其他证书错误均拒绝连接。
- 首版不提供忽略证书错误或为单个远程连接私下信任证书的能力。
- 远程服务端必须提供无需登录即可读取的兼容性握手，声明 API 版本和客户端相关能力，但不得泄露敏感配置。
- 客户端在显示远程认证界面前完成握手：接受声明为兼容的版本范围，并根据能力声明启用或禁用可选功能。
- 远程服务缺少握手或 API 版本不兼容时，客户端不得进入可能部分失效的主界面，必须明确提示升级服务端。
- 桌面客户端版本与远程服务端产品版本不要求完全一致；兼容判断以 API 版本和能力为准。
- 同一 API 主版本内，已经成为必需契约的端点、字段和行为必须保持向后兼容；新增可选能力通过能力声明渐进启用，不得借能力声明掩盖必需契约的破坏性变化。
- 只有无法在现有主版本内兼容的变更才提升 API 主版本。
- 引入新 API 主版本时，包含该主版本的首个稳定产品版本及其后一个稳定产品版本，桌面客户端必须同时支持新主版本和上一主版本；完成这两个连续稳定版本后才允许移除上一主版本。
- 发布门禁必须运行覆盖所有受支持 API 主版本的远程模式契约测试；任一受支持主版本失败时不得形成稳定 Release。
- 首次配置远程连接时，客户端必须依次完成地址规范化、HTTP 风险确认或 HTTPS 验证、兼容性握手，再进入远程注册/登录；验证失败不得保存为可用连接。
- 修改远程地址时必须先验证候选连接；验证失败保留原连接，验证成功后才原子替换并清除旧远程令牌和旧地址的风险确认。
- 移除唯一远程连接必须清除本机保存的地址、令牌、HTTP 风险确认和兼容性结果，但不得向远程服务端发出删除资料库的操作。
- 用户在远程模式中移除连接后必须返回模式选择页，禁止自动进入本地模式；在本地模式中移除连接不改变当前本地资料库。
- 本任务当前只做需求探索、研究和方案设计；方案评审并显式启动任务前不进入实现。

## Constraints Discovered from the Existing System

- 现有 Web/Docker/PWA 是已发布产品路径，桌面方案必须把它视为独立兼容面。
- AI 摘要、翻译、批量翻译与 OPML 进度依赖 SSE/NDJSON 增量传输，不能退化为完成后一次性返回。
- 现有 SQLite schema、自动迁移、JWT/API 契约和后台调度属于回归保护范围。
- Wails 3 当前仍为 Alpha；采用时必须锁定已验证版本并隔离框架依赖。
- 当前 Windows 后端测试已有两个平台相关失败，必须作为迁移前基线单独处理。

## Decisions Pending Design Review

- 是否接受 `design.md` 与 ADR-0002 的混合传输：普通 API 使用进程内模式感知 `http.Handler`，四类流式能力使用 Wails 任务事件，Web/PWA 保留 SSE/NDJSON，桌面不开放 localhost。
- 是否接受 `design.md` 与 ADR-0003 的更新恢复架构：签名 NSIS 安装版本化 payload，由稳定 launcher 在首次健康检查失败时恢复上一版本和升级前本地资料库。

## Acceptance Criteria

- [ ] 已形成有代码证据支持的当前项目背景、功能清单和架构边界说明。
- [ ] 已形成引用 Wails 官方一手资料的 Wails 3 技术研究记录。
- [ ] 已逐项确认客户端目标、兼容边界、平台范围、分发方式和迁移策略。
- [ ] 本地模式与远程模式均有独立、可验证的端到端验收场景。
- [ ] 切换模式后展示正确的独立内容资料库，且任一模式的写入不会改变另一模式的数据。
- [ ] 同一网络代理设置界面在本地模式直接读写本地服务、在远程模式在线直接读写远程服务；切换模式不会复制配置，远程保存失败不会产生本地伪成功状态。
- [ ] 本地与远程网络代理设置只影响各自后端的外部请求，不会改变桌面外壳、远程 API 连接或更新检查的网络行为。
- [ ] 桌面客户端始终只有一个本地资料库，且最多保存一个远程连接；替换或移除远程连接不会删除本地资料库。
- [ ] 选择本地模式后只使用本地注册/登录状态，选择远程模式后只使用远程认证状态；两套令牌不会串用。
- [ ] 桌面端本地与远程令牌分别存入 Windows 凭据管理器且不进入 WebView 持久存储；凭据写入失败不会降级为明文保存，登出只清理当前模式令牌，Web/PWA 认证行为保持不变。
- [ ] 有效令牌会在启动或模式切换时自动登录；只有后端明确拒绝认证才清除对应令牌，网络、安全、兼容性或临时服务故障均会保留令牌并显示正确错误状态。
- [ ] 首次启动要求选择模式，后续启动进入上次使用的模式，且用户可在应用内显式切换并持久化新选择。
- [ ] 主界面始终以文本显示当前本地或远程资料库来源；失败状态不会隐藏来源，切换完成前不会提前显示目标，HTTP 来源持续标识不安全。
- [ ] 远程服务不可用时不会展示本地资料库或改变启动模式，并可重试、修改连接或由用户显式切换到本地模式。
- [ ] Windows 10/11 x64 桌面产物能够安装、启动并完成本地模式与远程模式的端到端场景。
- [ ] NSIS 默认安装到当前用户的 `%LOCALAPPDATA%\Programs\Gist` 且无需管理员权限，不会创建计算机级状态；程序目录与用户数据目录的安装、升级和清理边界相互独立。
- [ ] 首版安装界面不能改变程序目录，后续升级、修复、回滚和卸载始终命中固定程序目录且不改变用户数据目录。
- [ ] NSIS 安装包可在 Windows 10/11 x64 完成安装、覆盖升级和卸载，且不会误删本地资料库。
- [ ] 后台更新检查不阻断主流程；新版本只在用户确认后下载并启动 NSIS，升级前完成优雅退出且保留用户数据。
- [ ] 前台任务存在时，更新包可以在首次确认后下载和验证，但安装器不会启动；任务结束后只有第二次确认才会退出并安装，暂缓操作没有任务或进程副作用。
- [ ] 已保存远程服务不受目标桌面版本支持时仍可完成桌面更新；安装后本地模式正常可用，进入远程模式时才提示 API 不适配且保留原连接状态。
- [ ] 覆盖升级前会保留上一正式版本和受保护的本地资料库恢复点；新版本首次启动或迁移验证失败会自动恢复两者，验证成功后的正常使用不会被任意降级机制倒退。
- [ ] 正式更新同时通过 Authenticode 与内置公钥验证；篡改、错签、缺签或测试通道产物均被拒绝。
- [ ] 同一稳定 Git Tag 产出的 NSIS、更新清单、服务端/Docker 与 Web/PWA 使用同一 Gist 产品版本并可追溯到同一提交；任一必需产物失败时不会形成部分稳定 Release。
- [ ] 正式客户端只发现 GitHub Releases 中的稳定版本，不会接收预发布或其他来源的产物。
- [ ] 自动更新检查遵守 24 小时间隔且不阻塞启动，手动检查可随时执行。
- [ ] 托盘或登录自启动状态发现更新时不会抢占主窗口；系统通知只引导用户打开确认流程，通知不可用时仍会在下次打开主窗口后提示且不会重复堆叠。
- [ ] 桌面本地资料库稳定使用 `%LOCALAPPDATA%\\Gist\\data`，同时现有服务端目录配置保持兼容。
- [ ] 桌面不会探测或共享现有服务端数据目录；远程使用无需迁移，显式完整备份/恢复可创建独立本地副本。
- [ ] 卸载默认保留全部用户数据；只有明确勾选并确认时才清理列明的数据，且不会删除 Gist 范围之外的文件或凭据。
- [ ] 用户可手动创建并恢复经过格式、完整性和版本验证的本地一致性备份；OPML 与完整备份用途清晰区分，远程资料库不受影响。
- [ ] 完整备份使用未被应用保存的用户密码加密，能够跨重装/设备完整还原本地状态，并拒绝错误密码、篡改或损坏数据。
- [ ] 完整恢复在确认后整体替换本地资料库，任何失败都自动回滚原数据，且不改变远程连接或远程资料库。
- [ ] 有前台任务时模式切换要求用户取消或留下；后台维护不阻塞切换，无法撤回的远程任务会准确说明状态。
- [ ] 窗口隐藏时，用户主动发起的前台任务完成或失败会发送不含敏感内容的 Windows 通知并可返回结果；窗口可见时不重复通知，后台维护始终不发送任务完成通知。
- [ ] 两种模式都使用内置前端，远程服务只作为 API/资料库；现有服务端 Web/PWA 仍可独立使用。
- [ ] 桌面端用户点击 HTTP/HTTPS 外链时由 Windows 默认浏览器打开，Wails 窗口不会离开内置应用；危险、非用户触发或打开失败的导航不会替换当前界面，Web/PWA 行为保持不变。
- [ ] 桌面端 OPML、完整备份/恢复和诊断包流程使用 Windows 原生文件对话框；取消操作无副作用，远程模式不泄露本机路径，Web/PWA 文件行为保持不变。
- [ ] 远程服务不可用时不展示持久化远程内容或接受离线写入，恢复操作不会混入本地资料库。
- [ ] 本地模式不开放网络监听端口，完整功能仅在桌面客户端内可用；现有 server/Docker 的 Web/LAN 行为不变。
- [ ] 桌面版以正式版本发布；Wails 依赖精确锁定，升级必须通过完整回归、安装和更新验证。
- [ ] 应用在远程模式中保持运行时，本地资料库仍按既有调度语义刷新，且不会重复调度远程资料库。
- [ ] 同一 Windows 用户重复启动 Gist 时始终只有一个进程、一套本地资料库访问和一套调度器；第二次启动会显示已有窗口，异常退出后的陈旧状态不会阻止重启。
- [ ] 点击关闭按钮会显示退出/托盘选择；退出路径完成优雅关闭，托盘路径继续本地刷新并可重新显示或完全退出。
- [ ] 存在前台任务时，任何完全退出入口都会要求用户明确取消任务；放弃退出或隐藏到托盘不会终止任务，无法撤回的远程任务会被准确说明。
- [ ] 用户可记住关闭行为，并能从设置中清除该偏好，使关闭按钮重新显示选择对话框。
- [ ] 登录自启动在新安装时默认关闭；开启后 Windows 登录只静默进入托盘并启动本地后台维护，关闭开关或卸载会移除应用创建的自启动项。
- [ ] 客户端不自动采集或上传遥测和崩溃报告；本地日志经过敏感信息脱敏并有界轮转，只有用户主动操作才生成不含资料库、凭据和文章内容的离线诊断包。
- [ ] 本地日志在超过 14 天或总量超过 50 MiB 时会优先删除最旧内容，且不会删除用户已经导出的诊断包。
- [ ] HTTP 远程连接在发送凭据前要求显式确认并持续标识风险；修改地址后必须重新确认。
- [ ] 首版只连接用户手动输入并完成验证的远程地址，不扫描、发现或自动选择局域网中的服务。
- [ ] 远程服务地址可使用显式端口和反向代理路径前缀，规范化后正确访问其 `/api`；内嵌凭据、查询参数、片段或路径逃逸会在发出凭据前被拒绝。
- [ ] 远程请求只自动跟随同一已确认地址及路径前缀内的有界重定向；跨地址、跨路径前缀、循环和 HTTPS 降级不会携带凭据继续请求。
- [ ] 无效 HTTPS 证书会在发送凭据前被拒绝，并显示可区分于普通网络故障的证书错误。

- [ ] 远程连接只在完整验证后保存；失败的替换保留原连接，移除连接只清理本机状态并按当前模式进入正确页面。
