# 规划 Wails 3 客户端兼容架构

> 状态：规划需求已确认；尚未启动实现
> 产品范围：Windows 10/11 x64 桌面客户端，同时保持现有 Web/PWA/Docker 契约

## 1. 文档职责与验收层级

本文只定义产品需求、约束和可验收结果，不规定模块、类型、文件、Child 执行顺序或具体实现。技术方案由 `design.md` 承担，任务拆分与证据追踪由 `implement.md` 承担。

验收分为两层：

- **规划完成条件（AC-PLAN-*）**：可以在编写产品代码前，以评审通过的需求、设计、研究证据和实施追踪表完成。
- **最终产品验收（AC-PROD-*）**：必须在后续 Child 实现后，以测试、构建产物、运行记录或人工场景证据完成；本规划任务不得将其标记为已实现。

## 2. 目标与非目标

### 2.1 目标

- **REQ-SCOPE-01**：规划基于 Wails 3 的正式 Windows 桌面客户端，最大限度复用现有 Gist 业务能力，并保持 Web/PWA/Docker 独立可用。
- **REQ-SCOPE-02**：本地模式与远程模式都是正式产品能力；本地模式无需预部署服务端，远程模式连接用户已部署的 Gist 服务。
- **REQ-SCOPE-03**：首个正式桌面交付范围为 Windows 10/11 x64、当前用户安装的 NSIS；Wails 的 Alpha 状态不改变正式发布、签名和支持要求。
- **REQ-SCOPE-04**：同一稳定 Git Tag 必须统一桌面、服务端/Docker 与 Web/PWA 的 Gist 产品版本和发布追溯关系。

### 2.2 非目标

- **REQ-NONGOAL-01**：首版不支持 Windows ARM64、macOS、Linux 桌面包或免安装便携包。
- **REQ-NONGOAL-02**：首版不支持多个本地资料库、多个远程连接、跨资料库聚合、资料库同步、远程离线副本或离线写入队列。
- **REQ-NONGOAL-03**：桌面本地模式不成为浏览器、回环地址或局域网可访问的第二个服务端；不直接打开或共享现有服务端数据目录。
- **REQ-NONGOAL-04**：桌面窗口不加载远程服务托管的 Web/PWA，也不替换现有服务端 Web/PWA 交付路径。
- **REQ-NONGOAL-05**：本规划任务不启动 Child、不修改产品代码，也不把 feasibility gate 的原型结果预判为成功。

## 3. 产品需求

### 3.1 既有产品兼容面（REQ-COMPAT）

- **REQ-COMPAT-01**：现有 HTTP API、JWT 行为、状态码、JSON、SSE/NDJSON、SQLite schema/迁移、定时调度、Web/PWA 认证存储、文件流程和 Docker/server 运行方式不得发生未经确认的破坏性变化。
- **REQ-COMPAT-02**：现有服务端继续支持 `./data`、`GIST_DATA_DIR`、`GIST_DB_PATH`、Web/LAN listener、Linux Docker 和 PWA；桌面固定目录不得改变这些契约。
- **REQ-COMPAT-03**：AI 摘要、翻译、批量翻译和 OPML 进度在 Web 与桌面均保持增量、取消和明确错误语义，不得退化为完成后一次性返回。
- **REQ-COMPAT-04**：Wails CLI、Go module、前端 runtime 及安全/打包依赖必须精确锁定为经联合验证的版本组合；禁止 `latest`、通配范围和未提交的解析结果。

### 3.2 资料库、设置与数据所有权（REQ-DATA）

- **REQ-DATA-01**：桌面只维护一个固定本地资料库和最多一个远程连接；内容、订阅、已读、收藏、设置、AI 缓存和认证状态按资料库隔离，不复制、合并或隐式同步。
- **REQ-DATA-02**：桌面本地资料库固定为当前用户 `%LOCALAPPDATA%\Gist\data`；程序、数据、日志、更新临时文件和 WebView 数据具有独立生命周期。
- **REQ-DATA-03**：网络代理设置属于当前资料库后端。同一界面在本地直接读写本地服务，在远程在线读写远程服务；切换时重新读取，失败不产生本地副本、离线队列或伪成功状态。
- **REQ-DATA-04**：资料库代理设置不得影响桌面外壳、桌面到远程 API 的连接或 GitHub 更新检查；首版不提供独立桌面连接代理。
- **REQ-DATA-05**：只要桌面进程运行，本地资料库保持既有定时刷新，即使 active repository 为远程；远程调度只由远程服务端负责。
- **REQ-DATA-06**：在创建本地 application Runtime、执行迁移或打开 SQLite 前，桌面必须取得独立的当前用户级、跨会话数据锁；锁定范围覆盖本地资料库迁移、访问和调度。锁失败不得创建 Runtime。
- **REQ-DATA-07**：Wails 单实例机制只负责第二次启动时唤醒/聚焦窗口，不承担数据互斥；数据锁必须能随进程退出释放，并允许异常终止后的后续正常启动。

### 3.3 认证与凭据（REQ-AUTH）

- **REQ-AUTH-01**：用户先选择本地或远程，再进入对应注册/登录；两套凭据、令牌和登录状态不得复用。
- **REQ-AUTH-02**：本地与远程令牌分别存入当前用户 Windows Credential Manager；远程项绑定规范化完整地址。令牌和密码不得持久化到 WebView `localStorage`、SQLite 设置或普通文件。
- **REQ-AUTH-03**：凭据存储失败时最多保留当前进程内存会话并明确提示，不得回退明文；登出只删除当前模式令牌。
- **REQ-AUTH-04**：已有令牌仅在目标安全校验和兼容性握手后验证。只有后端明确判定令牌无效、过期或撤销时才清除；网络、TLS、兼容性或临时故障保留令牌。
- **REQ-AUTH-05**：Web/PWA 保持现有浏览器认证存储和 API 行为。

### 3.4 active repository 与 candidate transition（REQ-MODE）

- **REQ-MODE-01**：模式状态必须同时表达一个已提交的 **active repository** 与至多一个未提交的 **candidate transition**；验证候选期间 active repository、启动偏好、来源标识和可用内容保持不变。
- **REQ-MODE-02**：candidate transition 依次完成目标安全、兼容性和认证验证后才原子提交；失败或取消只丢弃 candidate，不改变 active repository。
- **REQ-MODE-03**：每次成功提交生成严格递增的 **generation**；请求、流式事件、缓存写入和通知结果必须绑定 generation，旧 generation 的迟到结果不得进入新 active repository 的界面。
- **REQ-MODE-04**：首次启动显示模式选择；成功选择后保存启动模式。应用提供显式切换入口，远程启动失败时停留在远程错误状态并提供重试、修改连接和显式切到本地，禁止自动回退。
- **REQ-MODE-05**：主界面始终用可读文本显示 active repository 来源；远程显示规范化主机、显式端口和路径前缀，HTTP 同时持续显示不安全标识。candidate 未提交前不得替换该标识。
- **REQ-MODE-06**：切换提交必须阻止旧资料库请求/结果继续进入 UI，并清除所有资料库相关查询、内存状态、选择、滚动和持久前端缓存；相同实体 ID 不得导致串库。

### 3.5 远程地址、安全与兼容性（REQ-REMOTE）

- **REQ-REMOTE-01**：远程地址只由用户手动输入，不扫描或自动发现；仅接受含主机的 HTTP(S)，可含端口和路径前缀，拒绝 userinfo、query、fragment 和路径逃逸。
- **REQ-REMOTE-02**：地址变化后重新执行规范化、安全校验和握手。HTTP 在发送凭据前要求明确风险确认；HTTPS 使用 Windows 系统信任链且不提供证书绕过。
- **REQ-REMOTE-03**：携带凭据的请求只可在相同 scheme/host/port 且仍位于已确认路径前缀内进行有界重定向；跨边界、循环、超限或 HTTPS 降级必须停止且不得泄露凭据。
- **REQ-REMOTE-04**：修改连接时先验证 candidate，成功后才原子替换并清理旧地址关联的令牌/确认；失败保留原连接。移除连接只清理本机状态，不删除远程数据，也不自动进入本地。
- **REQ-REMOTE-05**：远程不可用时不得以持久缓存伪装在线资料库，也不得接受离线写入。
- **REQ-REMOTE-06**：公开兼容性握手不得泄露敏感配置；在任何凭据发送前完成。
- **REQ-REMOTE-07**：`api.major` 定义该 major 的基础必需契约；同一 major 的必需端点、字段和行为保持兼容。**optional capability** 只声明可选功能，缺失时仅禁用对应可选功能，绝不能掩盖基础必需契约的缺失或破坏。
- **REQ-REMOTE-08**：客户端维护显式 supported-major 集合。引入新 major 的首个稳定版本及其后一个稳定版本必须同时支持新旧 major，并对每个受支持 major 运行远程契约测试；产品版本号不作为兼容相等条件。

### 3.6 流式任务与 OperationCoordinator（REQ-TASK）

- **REQ-TASK-01**：四类流式任务必须保留事件顺序、增量内容、缓存命中、进度、取消、错误和唯一终态；文本/条目不得丢失或重排，进度可合并为最新值。
- **REQ-TASK-02**：流式传输必须具有有界背压和连续 ACK 语义；接收方只确认无缺口的最高连续 sequence，乱序、重复、缺口、reload 和失联不得造成误确认、无界内存或跨 generation 投递。
- **REQ-TASK-03**：模式切换、完整退出、恢复和更新安装由同一 **OperationCoordinator** 查询并协调用户前台任务；任一入口不得绕过任务门禁。
- **REQ-TASK-04**：存在前台任务时，切换/退出必须提供“取消任务并继续”和“留在当前状态”；隐藏到托盘不取消任务。无法撤回的远程任务必须如实说明仍可能继续。
- **REQ-TASK-05**：后台维护不阻塞模式切换；更新可在首次确认后下载验证，但有前台任务时不得启动安装器，任务结束后仍需第二次明确确认。

### 3.7 Windows 生命周期、通知与外链（REQ-SHELL）

- **REQ-SHELL-01**：同一 Windows 用户重复启动时只保留一个 Runtime、迁移和调度器。第二次启动在取得数据锁失败后通过不访问资料库的激活 IPC 请求已有实例显示窗口；同一交互会话内应唤醒并聚焦，跨 Terminal Services session 无法安全聚焦时显示明确的“另一会话正在使用资料库”占用状态。
- **REQ-SHELL-02**：关闭按钮提供完全退出/隐藏托盘及“记住选择”；设置可重置。所有完全退出入口使用 OperationCoordinator，退出优雅关闭，隐藏后本地维护和任务继续。
- **REQ-SHELL-03**：当前用户级登录自启动默认关闭；开启后静默进入托盘并启动本地维护，不因远程错误弹窗；关闭开关和卸载清理应用创建的自启动项。
- **REQ-SHELL-04**：隐藏时，用户发起任务的成功/失败及可用更新可发送不含敏感内容的 Windows 通知；可见时不重复，后台维护不发送任务通知。通知失败时结果/更新状态保留在应用内。
- **REQ-SHELL-05**：点击通知必须激活已有实例并恢复相关任务或更新上下文；原页面不可恢复时显示明确结果，不得静默丢失。
- **REQ-SHELL-06**：Wails 窗口只导航内置资源。用户触发的 HTTP(S) 外链交给默认浏览器；非用户触发及 `javascript:`、`data:`、`file:` 等协议被阻止，失败不替换当前界面。Web/PWA 保持浏览器行为。

### 3.8 原生文件流程、备份与恢复（REQ-FILE）

- **REQ-FILE-01**：桌面 OPML 导入/导出、完整备份/恢复和诊断包导出使用 Windows 原生对话框；取消无副作用。远程导入只上传内容，远程端不得获得本机路径。
- **REQ-FILE-02**：用户主动的 local-only 完整备份覆盖可还原的账号、订阅、文章、阅读/收藏、设置、AI/代理凭据、缓存和图标；排除远程连接/令牌、日志、更新和远程资料库，并与 OPML 明确区分。
- **REQ-FILE-03**：每次备份使用用户提供且应用不保存/同步/找回的独立密码，以跨重装/设备的加密认证格式和一致性快照生成。
- **REQ-FILE-04**：恢复在修改 active repository 前完成解密、认证、格式、完整性、兼容版本、资源上限和路径安全验证，再明确确认整体替换；不提供合并恢复。
- **REQ-FILE-05**：恢复前创建受保护回滚快照并停止相关访问；替换或重开失败自动恢复原资料库，远程连接、凭据和资料库不变。

### 3.9 安装、更新、回滚与发布（REQ-RELEASE）

- **REQ-RELEASE-01**：NSIS 固定安装到 `%LOCALAPPDATA%\Programs\Gist`，不提供目录选择、不要求管理员权限、不创建计算机级状态；安装、修复、覆盖升级、回滚和卸载命中同一路径。
- **REQ-RELEASE-02**：卸载默认保留用户数据；仅当用户明确勾选并确认时删除列明的 Gist 自有目录、凭据和自启动项，不得越界清理。
- **REQ-RELEASE-03**：正式客户端仅从现有 GitHub Releases 接受稳定版本；启动后距上次检查超过 24 小时才自动检查，持续运行每 24 小时检查，手动检查始终可用，失败不阻断产品。
- **REQ-RELEASE-04**：更新需要“允许下载验证”和“退出并启动安装器”两次独立确认；暂缓不取消任务、不退出、不损坏已验证待安装状态。
- **REQ-RELEASE-05**：安装前保留上一正式版本和一致性本地数据恢复点；新版本首次启动、迁移或完整性健康检查失败时自动恢复两者。提交成功后不提供任意降级；回滚失败保留材料并停止循环启动。
- **REQ-RELEASE-06**：安装包同时通过 Authenticode 与内置公钥签名验证；错签、缺签、篡改和测试通道产物均失败关闭。发布包含私钥保管、CI 权限、签名失败关闭和公钥轮换约束。
- **REQ-RELEASE-07**：一个稳定 Release 同步产生同版本 NSIS、签名清单、server/Docker 和 Web/PWA，并追溯同一提交；任一必需资产、测试或签名失败不得形成部分稳定 Release。
- **REQ-RELEASE-08**：已保存远程服务与新客户端 API major 不兼容不得阻止下载安装；升级后仅在进入/重试远程时提示，并保留连接、令牌与 HTTP 确认，本地模式不受限。

### 3.10 隐私、日志与诊断（REQ-PRIVACY）

- **REQ-PRIVACY-01**：首版不自动采集或上传遥测、诊断、日志或崩溃报告，不内置默认启用的第三方遥测 SDK。
- **REQ-PRIVACY-02**：日志只保存在 `%LOCALAPPDATA%\Gist\logs`，写入前移除密码、令牌、密钥、备份密码、正文及可恢复内容；最多保留 14 天且目录最多 50 MiB，任一上限触发后从最旧项清理。
- **REQ-PRIVACY-03**：诊断包仅由用户主动导出到自选位置，生成过程不上传；只含版本、Windows/WebView2 环境摘要和脱敏日志，不含资料库、备份、凭据或正文。已导出文件不受日志轮转删除。

### 3.11 feasibility gates（REQ-GATE）

以下是进入依赖其结论的产品实现前必须独立完成并留存证据的硬门禁；原型只验证技术可行性，不替代最终产品 AC：

- **REQ-GATE-01**：Wails AssetServer 对普通 body、分块 body、multipart/upload、响应 status/header/body 和取消传播的完整 tracer bullet。
- **REQ-GATE-02**：乱序/重复/缺口事件、最高连续 ACK、有界窗口、背压、reload/失联和 generation 隔离的流式原型。
- **REQ-GATE-03**：窗口关闭 hook、托盘退出、系统退出路径均能进入 OperationCoordinator 且不会重复或绕过协调的生命周期原型。
- **REQ-GATE-04**：Windows 通知在已有实例、隐藏窗口和冷/热激活条件下的激活路由原型，证明可恢复明确上下文或提供受控降级。
- **REQ-GATE-05**：稳定 launcher 与版本化 payload 的安装器交接、事务标识、首次启动健康提交、超时/崩溃回滚和失败材料保留原型。

## 4. 规划完成条件

- [ ] **AC-PLAN-01**：现有产品背景、用户能力、代码边界和 Web/PWA/Docker 兼容面均有仓库证据，评审结论无未解释冲突。
- [ ] **AC-PLAN-02**：Wails 3 能力、Alpha 风险、锁定版本组合及五项 REQ-GATE 均有一手资料或明确待原型验证项，未知项未被写成既成事实。
- [ ] **AC-PLAN-03**：本文每个需求域和所有 AC 具有稳定 ID；`design.md` 能逐项引用需求，`implement.md` 维护“AC → 设计 → Child → 验证/证据”追踪且无孤立产品 AC。
- [ ] **AC-PLAN-04**：规划明确区分 active repository/candidate transition/generation、optional capability/API major 基础必需契约、独立数据锁/Wails 窗口唤醒和 OperationCoordinator 的职责。
- [ ] **AC-PLAN-05**：约 13–15 个小型 Child 的边界、依赖、feasibility gate 和回滚点已定义；每个产品 AC 至少映射到一个负责 Child，跨 Child AC 指定最终集成责任。
- [ ] **AC-PLAN-06**：用户已评审并接受 `prd.md`、`design.md`、ADR 与 `implement.md` 的完整需求边界和决策；在此之前不启动父任务或 Child。

## 5. 最终产品验收

### 5.1 兼容、模式与数据

- [ ] **AC-PROD-01**：Web/PWA/Docker/server 的 API、认证、流式、文件、调度、数据目录和部署回归套件保持通过，桌面进程无 localhost/LAN listener。（REQ-COMPAT-01~04、REQ-NONGOAL-03）
- [ ] **AC-PROD-02**：本地与远程各自完成注册/登录、订阅、文章、已读、收藏、设置、AI 与 OPML 端到端场景；使用相同实体 ID 的两套 fixture 连续切换后，DOM、查询缓存、前端持久状态、图片和写入只属于 active repository。（REQ-DATA-01、REQ-MODE-01~06）
- [ ] **AC-PROD-03**：首次选择、重启恢复、候选失败/取消、远程错误、显式切换和 generation 迟到结果场景均保持正确 active 来源；candidate 未提交前 UI 与启动偏好不变化。（REQ-MODE-01~05）
- [ ] **AC-PROD-04**：代理设置在两种模式正确直达各自后端，切换不复制，远程失败不伪成功，且桌面远程连接和更新检查不受资料库代理影响。（REQ-DATA-03~04）

## 6. 约束与已知基线

- 当前 Windows 后端测试存在两个平台相关失败，必须作为迁移前独立基线处理，不能归因于 Wails 或由桌面实现掩盖。
- 所有产品 AC 必须由后续具体 Child 引用；单元测试、集成测试、E2E、构建/签名记录或明确人工场景至少提供一种可复核证据。
- 需求变更不得通过 capability、默认回退、缓存展示、兼容别名或临时 shim 静默缩小已确认范围。
